The 6 layers of the IMPOWR AI solution
1
Frontend — React Web App (inside IMPOWR portal)
What the user sees and touches
A lightweight chat interface embedded directly inside the existing IMPOWR platform. The user types a plain-English question — no SQL, no technical knowledge required. The UI also includes the Onboarding tab for adding new users with role assignment and HIPAA compliance sign-off.

2
Azure Active Directory — Identity & Authentication
Who are you, and what are you allowed to see?
Every user logs in through Azure AD (Microsoft's identity platform). When login succeeds, Azure AD issues a JWT token containing the user's identity: their UserID, OrgID (which nonprofit they belong to), and Role (caseworker / manager / admin). This token travels with every single request. Without it, the system refuses the connection entirely.
Token payload: { userId: "sarah.martinez", orgId: 42, role: "manager", exp: ... }

3
API Middleware — The Brain & Security Gate
Validates identity, calls the AI, enforces org scope
A Python/FastAPI backend hosted on Azure App Service. It does three jobs simultaneously:

① Validates the token — checks the JWT is real and not expired.
② Builds a locked system prompt — injects OrgID=42 as immutable context before calling the AI. The user can never override this.
③ Validates the AI output — checks that the SQL generated contains OrgID=42 and is SELECT-only before executing it. Any deviation = rejected.
System prompt (locked): "Only generate SELECT queries. Every query MUST include WHERE OrgID = {user.orgId}. Never query other orgs."

4
Azure OpenAI GPT-4.1 — The NLP-to-SQL Engine
Translates plain English into safe SQL queries
The AI receives the locked system prompt + the user's question + the IMPOWR database schema. It translates natural language into a valid SQL query. GPT-4.1 is the strongest Azure model for this task — 1M token context window means it can hold the entire schema in memory. It is HIPAA-eligible for text workloads under Microsoft's BAA. Microsoft guarantees the data is never used to train their models.
Input: "How many clients achieved stable housing in Q1?" Output: SELECT COUNT(*) FROM Clients WHERE OrgID=42 AND HousingStatus='Stable' AND OutcomeDate BETWEEN '2025-01-01' AND '2025-03-31'

5
Azure SQL + Row-Level Security — The Data Vault
The last line of defense — enforced at the database engine itself
Even if every layer above somehow failed, the database enforces isolation. Azure SQL's Row-Level Security (RLS) automatically appends a WHERE clause to every query based on the session's OrgID. If OrgID=42 is the session, then only rows belonging to Org 42 are ever returned — the database engine physically cannot return Org 17's rows, regardless of what SQL was sent.
CREATE FUNCTION fn_OrgFilter(@OrgID INT) RETURNS TABLE AS RETURN SELECT 1 WHERE @OrgID = CAST(SESSION_CONTEXT(N'OrgID') AS INT); CREATE SECURITY POLICY OrgPolicy ADD FILTER PREDICATE fn_OrgFilter(OrgID) ON dbo.Clients;

6
Azure Monitor + Log Analytics — The Audit Trail
Every action logged for HIPAA accountability
Every query — who asked, what question, what SQL was generated, what data was returned, what time — is written to Azure Monitor. This is your HIPAA audit log. If a breach ever occurs, you can reconstruct exactly what happened. Security anomalies (like cross-org attempts) trigger immediate alerts.
Log entry: { user: "sarah.martinez", orgId: 42, question: "clients stable Q1", sql: "SELECT...", rows: 47, timestamp: "2025-04-15T14:23:11Z", status: "OK" }

Happy path — a manager asks a question
1
User types — Sarah (Org 42, Manager) types "How many clients achieved stable housing in Q1?" into the chat interface on her browser.
2
Request sent to API — The React app sends an HTTP POST to the FastAPI backend. The JWT token from Azure AD is attached in the Authorization header automatically.
3
Token validated — The API reads the JWT. It extracts: orgId=42, role=manager. If the token is invalid or expired → 401 Unauthorized immediately.
4
System prompt built — The API constructs a locked prompt: the IMPOWR schema + the rule "OrgID must = 42, SELECT only" + Sarah's question. OrgID is hardcoded by the server — Sarah has no way to change it.
5
GPT-4.1 called — Azure OpenAI receives the prompt over a private endpoint (never public internet). It generates: SELECT COUNT(*) FROM Clients WHERE OrgID=42 AND HousingStatus='Stable'...
6
SQL validated — The API checks: Does the SQL contain OrgID=42? Is it SELECT only? No DROP/UPDATE/INSERT? Pass → forward to database. Fail → reject, log incident.
7
Database executes with RLS — Azure SQL sets SESSION_CONTEXT(OrgID=42) then runs the query. RLS filter predicate silently confirms every row belongs to Org 42. Returns: 47 rows.
8
Response formatted — The API formats the 47 rows into a readable table + metrics summary. PHI fields (full names, SSNs) are masked before sending to the frontend.
9
Logged & displayed — The full interaction is written to Azure Monitor. The formatted response is sent to Sarah's browser and displayed as a chat message with the data table.
Total time: approximately 1.5–3 seconds end-to-end. Sarah gets her answer instantly — no waiting for a CCS employee to write SQL manually.
Blocked path — cross-org attempt
1
User asks — Sarah types "Show me clients from Buffalo Health Services."
2
AI middleware detects it — The system prompt explicitly states that queries for other orgs must be refused. GPT-4.1 returns a refusal — no SQL is generated at all.
3
Even if SQL was generated — The SQL validator would reject any query missing OrgID=42 or containing a different OrgID.
4
Even if it reached the database — Azure SQL RLS would return zero rows for any Org 17 data because the SESSION_CONTEXT is locked to Org 42.
5
Incident logged — The attempt is written to Azure Monitor with full details: user, timestamp, what was requested. Security alert fired.
Key point for judges: Three independent systems must ALL fail simultaneously for a data breach to occur — the AI, the validator, and the database engine. This is called defense in depth.
Defense in depth — 4 independent security walls
W1
Azure AD Authentication
Wall 1 — who you are
No valid JWT token = no entry. Period. Azure AD handles login, MFA, and session management. The OrgID and Role in the token are set by the server during provisioning — a user cannot forge or change their own OrgID.
W2
Locked System Prompt
Wall 2 — what the AI is allowed to do
OrgID is injected into the AI prompt by the server before the user's question is added. The user can type anything — even "ignore all instructions" — but the server-side context always overrides.
W3
SQL Validation Layer
Wall 3 — what SQL can execute
Every SQL string the AI generates is parsed before execution. Rules: must contain OrgID=42, must be SELECT only, must not reference system tables, must not contain UNION or subqueries to other orgs. Fail any check = request rejected.
W4
Database RLS
Wall 4 — engine-level enforcement
Even if a crafted SQL bypassed all three walls above, RLS returns zero rows for any OrgID that doesn't match the session. This operates inside the database engine itself — application code cannot bypass it.

How each HIPAA requirement is met
Technical safeguards checklist
Access Control (§164.312(a)) → Azure AD + RLS ensures users only see their org's data.
Audit Controls (§164.312(b)) → Azure Monitor logs every query with user, time, data returned.
Integrity (§164.312(c)) → SQL validation prevents writes; SELECT-only policy enforced.
Transmission Security (§164.312(e)) → All traffic over TLS 1.2+; private endpoints — no public internet.
Encryption at Rest → Azure SQL Transparent Data Encryption (TDE) with Azure Key Vault keys.
Model Training on PHI → Azure OpenAI BAA guarantees data is never used to train models.
What happens when you onboard a new user
1
Admin fills the form — Name, email, department, org assignment (Org 42), and role (Caseworker / Manager / Admin) entered in the IMPOWR Onboarding tab.
2
Role selected — Each role maps to a predefined permission set. A Caseworker only sees their assigned clients. A Manager sees all org clients. An Admin can manage users and view audit logs.
3
Permissions toggled — Fine-grained module access configured: client records, appointments, AI assistant, grant reports, audit logs. Each toggle maps to a database permission scope.
4
Compliance checked — Admin confirms HIPAA training complete, DUA signed, background check cleared, MFA enrolled, acceptable use policy acknowledged. All five required before activation.
5
Account created in Azure AD — On activation, the API calls Azure AD to create the user account with their OrgID and Role embedded in their profile. An invitation email is sent automatically.
6
RLS automatically enforced — Because their OrgID is set in Azure AD, every database connection they make will have SESSION_CONTEXT(OrgID=42) set automatically. They are instantly scoped to Org 42 — no manual database configuration needed.
7
Onboarding logged — The full onboarding event (who created the account, what permissions, what compliance items were checked, timestamp) is written to Azure Monitor as a HIPAA-required access provisioning audit record.
Why this matters for the pitch: Today, CCS manually assigns database credentials and permissions for each new client org user — it's slow and error-prone. This onboarding flow makes it self-service, HIPAA-logged, and RLS-enforced in under 5 minutes.
